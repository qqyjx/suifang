# 周一交付体验版：链路打通 + 鲁棒性补齐 + 体验/正式版区分

## Context（为什么做这件事）

用户用 G band（第三方 BLE 应用）成功绑定了一块 Veepoo 兼容手表 **S101 / `FA:BA:94:8A:70:75`**（已充电、密钥认证通过），证明硬件本身没问题。

但目前从手表 → 小程序 → `dc.ncrc.org.cn/api2` → 六元 MySQL 这条真机链路还有两个阻塞：

1. **生产 `wearable_device` 表只登记了 demo 的 3 台 VP-W680**（id=1/2/3，MAC 前缀 B8:27:EB），还没有 S101 的归属行。真机上传虽然能落到 `wearable_device_data`，但 `deviceId=4` 是个孤儿——没有对应的设备元信息。
2. **`dataStorage.ts:215` 把 `deviceId: 4` 写死了**（注释里也明确写着「TODO: 后续按 BLE MAC 动态映射」）。一旦未来接第二只表就要改代码，对脚手架式接入的医院诊室部署不友好。

目标：把 S101 注册成 deviceId=4，并把硬编码改为按 BLE 设备签名（name + MAC/UUID）向服务端动态查询/注册——后续每只新表第一次被某位患者的手机扫到时，小程序就**通过 HTTPS** 在中央数据库里建一行并取回自己的 deviceId，工程师无需介入。

### 架构边界（先讲清楚 BLE 到底连谁）

```
[患者 A 床旁/家里]                                     [公司机房]
S101 手表 A ──BLE──> 患者 A 的手机（小程序）───┐
                                              ├─ HTTPS POST ─> dc.ncrc.org.cn/api2
S101 手表 B ──BLE──> 患者 B 的手机（小程序）───┤              ↓ 反向代理
                                              │       192.168.4.104  Node 服务
S101 手表 N ──BLE──> 患者 N 的手机（小程序）───┘              ↓ SQL
                                                      192.168.4.174  六元 MySQL
                                                      ├ wearable_device（设备名册）
                                                      └ wearable_device_data
```

- **BLE 永远是「手表 ↔ 患者身边那部手机」的短距离链路**，公司服务器没有蓝牙硬件也不在患者旁边，**绝不直接收 BLE**（README「为什么必须用手机/小程序」段落已说明）。
- 「服务端自动建行」= 患者手机的小程序通过 **HTTPS** 调 `/api/device/register` 把 MAC 上报，**不是手表自己握手到服务器**。
- 交付到医院时：每位患者一只表 + 一部手机（或医院统一发放的平板）装小程序；100 位患者 = 100 条独立的 BLE 链路 + 100 个 HTTPS 上行，全部汇到这一台中央服务器。这个中央表 `wearable_device` 就是「全院手表名册」。

### 绑定 key 设计与边界（重要——避免后续重复踩坑）

`wearable_device.device_sign` 的取值固定为：
```
device_sign = `${bleInfo.name}_${bleInfo.deviceId}`
            ↑                ↑
            BLE 广播名(S101)  Android: 真 MAC（FA:BA:94:8A:70:75）
                             iOS:     系统给的 36 字符 UUID（不是 MAC）
```

**为什么不用纯 MAC**：iOS 从 BLE 协议层就对所有 APP 屏蔽 MAC，任何 SDK（包括 Veepoo / G band）都拿不到——`bleInfo.deviceId` 在 iOS 是 `peripheralIdentifier` UUID，在 Android 才是 MAC。这是苹果系统级限制无法绕开。

**这个 key 能正确处理**：
- 患者一直用同一部手机 + 同一只表 → 永远同一个 device_sign，数据持续续流
- 同一部手机换新表 → 新 sign 触发服务端 INSERT 新行，分配新 deviceId
- 跨部门同型号手表混用：name 相同但 MAC/UUID 不同 → 各自独立 deviceId

**已知边界（接受、不在本次解决）**：
- iOS 患者换新 iPhone → 同一只表会注册成第二个 deviceId（旧数据不丢，但不再续传），需护士在六元后台手工 merge 两条记录。**临床场景下患者换手机频率极低**，故选择接受。
- 同一只表先后给两个患者用 → 系统会归到同一个 deviceId（`wearable_device_data` 按 deviceId 主键 UPSERT 会污染数据）。患者发表流程上不应允许，靠管理流程兜底，不在代码层防御。

**未来如需"跨平台跨手机绝对唯一"**：在 BLE 密钥认证成功后多调一次 vp_sdk 的设备信息接口读取手表硬件 SN，把 sign 改为 `name_SN_<硬件序列号>`。改动只需替换 `dataStorage.resolveDeviceId()` 里构造 sign 的那一行，服务端无需改动。

### Step 4：小程序端做动态 deviceId 解析

```ts
private deviceIdCache: number | null = null;

private async resolveDeviceId(): Promise<number> {
  if (this.deviceIdCache !== null) return this.deviceIdCache;
  const bleInfo: any = wx.getStorageSync('bleInfo');
  if (!bleInfo || !bleInfo.deviceId) return 4;            // 兜底：没连过 BLE 时仍走 4
  const sign = `${bleInfo.name || 'unknown'}_${bleInfo.deviceId}`;
  return new Promise((resolve) => {
    wx.request({
      url: `${WSL_SERVER_URL}/api/device/register`,
      method: 'POST',
      data: { deviceSign: sign, type: 1 },
      success: (res: any) => {
        const id = res?.data?.deviceId;
        if (typeof id === 'number') { this.deviceIdCache = id; resolve(id); }
        else resolve(4);
      },
      fail: () => resolve(4),
    });
  });
}
```

### Step 5：断网重传队列（医院网络不稳是常态）

**修改文件**：`code/demo/WeiXinSDKTSDemo/miniprogram/services/dataStorage.ts`

当前 `syncToServer` 在 `wx.request.fail` 时只 `console.log`，数据直接丢——这在病房 WiFi 抖动场景会丢分钟级心率/血氧。

新增 pending queue：

```ts
// 类内字段
private static PENDING_KEY = 'sync_pending_queue';
private static MAX_QUEUE = 500;  // 防爆，超出 LRU 丢最旧的

// 失败入队（在 syncToServer 的 fail 回调里）
private enqueuePending(payload: any): void {
  const queue: any[] = wx.getStorageSync(DataStorage.PENDING_KEY) || [];
  queue.push({ ...payload, _enqueuedAt: Date.now() });
  if (queue.length > DataStorage.MAX_QUEUE) queue.splice(0, queue.length - DataStorage.MAX_QUEUE);
  wx.setStorageSync(DataStorage.PENDING_KEY, queue);
}

// 触发刷队列
public async flushPending(): Promise<{ ok: number, fail: number }> {
  const queue: any[] = wx.getStorageSync(DataStorage.PENDING_KEY) || [];
  if (queue.length === 0) return { ok: 0, fail: 0 };
  const remaining: any[] = [];
  let ok = 0;
  for (const item of queue) {
    try {
      await this.postOnce(item);   // 抽出 wx.request 包装成 Promise
      ok++;
    } catch {
      remaining.push(item);
    }
  }
  wx.setStorageSync(DataStorage.PENDING_KEY, remaining);
  return { ok, fail: remaining.length };
}
```

```ts
    // 顺便刷 Step 5 的 pending 队列
    dataStorage.flushPending().then(r => console.log('[Sync] 补传', r));
  },
});
```

**重连失败处理**：本次不做指数退避，重连失败就保持断开状态（用户下次进入连接页会手动重连）。医院场景下手表丢失/电量耗尽是真实情况，过度自动重连反而耗手机电。

### Step 7：BLE 扫描设备过滤（医院多设备场景）

**修改文件**：`code/demo/WeiXinSDKTSDemo/miniprogram/pages/bleConnection/index.ts:60-66`

当前 `onBluetoothDeviceFound` 直接把所有扫到的设备塞进 `devicesList` 然后按 RSSI 排序——病房 100 块表 + 患者其他蓝牙设备时列表会爆。

加两层过滤：

```ts
// 配置：兼容机型白名单（VP-W680 是 demo，S101/VPR04 是真机型号）
private static SUPPORTED_NAME_PREFIXES = ['VP-', 'S101', 'VPR'];
private static MIN_RSSI = -75;  // dBm，约 3 米内可靠连

// 在 onBluetoothDeviceFound 回调里
const filtered = devices.filter((d: any) => {
  if (!d.name) return false;
  if (d.RSSI < BleConnectionPage.MIN_RSSI) return false;
  return BleConnectionPage.SUPPORTED_NAME_PREFIXES.some(p => d.name.startsWith(p));
});
```

并加扫描 30 秒自动停止：

```ts
private scanTimer: number = 0;
startScan() {
  // ... existing
  this.scanTimer = setTimeout(() => {
    veepooBle.veepooWeiXinSDKStopSearchBleManager(() => {});
    wx.showToast({ title: '扫描结束，未找到请重试', icon: 'none' });
  }, 30000) as unknown as number;
}
onUnload() {
  if (this.scanTimer) clearTimeout(this.scanTimer);
}
```

**白名单可配置**：把 `SUPPORTED_NAME_PREFIXES` 提取到 `services/env.ts`（Step 8 新建）的常量，便于后续不同医院自定义。
