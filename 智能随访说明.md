# 智能随访系统 — 可穿戴设备体征数据采集与结构化处理平台

## 一、项目概述

本项目是"智能穿戴设备集成技术方案"的核心软件部分，实现了从可穿戴医疗设备到研究队列数据库的完整数据链路。系统通过微信小程序 BLE SDK 和 Android BLE SDK 两个客户端，对接多款具备二类医疗器械注册证资质的智能硬件设备，自动采集 11 类人体体征数据，经 JSON Schema 校验和 FHIR Resources 标准化映射后，上传至 MySQL 研究队列数据库，支撑临床随访研究的数据管理需求。

**考核指标达成情况**：已集成 2 款具备二类医疗器械注册证资质的 Veepoo 可穿戴智能硬件设备（达成 ≥2 款的要求），采集信息经结构化分析后上传至研究队列。

**两个 SDK 的分工**：

| SDK | 路径 | 平台 | 职责 |
|-----|------|------|------|
| WeChat Mini Program BLE SDK | `/home/qq/WeChat_Mini_Program_Ble_SDK` | 微信小程序（iOS/Android/鸿蒙） | BLE 设备连接、数据采集、结构化处理、HTTP 同步 |
| Android BLE SDK | `/home/qq/Android_Ble_SDK` | Android 原生应用 | BLE 设备连接、原生 Android 端数据采集 |

两者共享同一套设备协议层（Veepoo SDK）和后端数据库，采集的数据最终汇入同一个研究队列数据库。

### 各层说明

**设备层**：2 款 Veepoo 二类医疗器械通过 BLE 协议与客户端通信，使用私有协议由 `vp_sdk`（378KB 编译后 JS）封装；杰理芯片设备由 `jieli_sdk` 处理 OTA 和表盘传输。

**采集层**：微信小程序端提供 58 个功能页面，覆盖心率测量、血氧监测、ECG 测量、血糖检测、睡眠分析、数据导出等全部功能。数据采集后先写入本地文件系统缓存，再通过 HTTP 同步服务上传。Android SDK 提供等效的原生端采集能力。

**结构化处理层**：
- JSON Schema 校验：所有体征数据按 `healthData.ts` 中定义的 11 个 TypeScript 接口校验字段完整性和类型正确性
- FHIR Resources 映射：将体征数据映射为 HL7 FHIR 标准资源（Observation、DiagnosticReport），确保医疗数据互操作性
- 传输通道：支持 HTTPS（同步上传）和 MQTT（实时流式传输）两种方式
- 手动数据：支持通过 PaddleOCR-VL 识别纸质报告 + Qwen3 智能解析，处理非数字化来源的体征数据

**数据库层**：MySQL 8.0 研究队列数据库，9 张表覆盖队列管理、受试者信息、设备溯源、体征数据、同步审计、结构化报告全链路。

## 四、数据采集能力（11 类体征数据）

| 序号 | 数据类型 | TypeScript 接口 | 关键字段 | 单位 | 采集频率 | 数据来源设备 |
|------|---------|----------------|---------|------|---------|------------|
| 1 | 心率 | `HeartRate` | `heartRate`, `heartState` | bpm | 实时/每分钟 | R04 戒指、AMOLED 手表 |
| 2 | 血氧饱和度 | `BloodOxygen` | `spo2` | % | 每 2 小时 | R04 戒指、AMOLED 手表 |
| 3 | 血压 | `BloodPressure` | `systolic`, `diastolic`, `riskLevel` | mmHg | 早/中/晚 | AMOLED 手表 |
| 4 | 体温 | `Temperature` | `temperature`, `skinTemperature` | °C | 每 4 小时 | AMOLED 手表 |
| 5 | 心电图 (ECG) | `ECG` | `ecgData[]`, `leadType` | mV | 按需测量 | R04 戒指（单导联） |
| 6 | 血糖 | `BloodGlucose` | `bloodGlucose`, `riskLevel` | mmol/L | 餐前/餐后 | AMOLED 手表 |
| 7 | 血液成分 | `BloodLiquid` | `uricAcid`, `bloodLipid` | 多种 | 每日 | AMOLED 手表 |
| 8 | 身体成分 | `BodyComposition` | `bmi`, `bodyFat`, `muscleMass` | 多种 | 每日 | AMOLED 手表 |
| 9 | 步数 | `Step` | `stepCount`, `distance`, `calories` | 步/km/kcal | 每日汇总 | R04 戒指、AMOLED 手表 |
| 10 | 睡眠 | `Sleep` | `deepSleep`, `lightSleep`, `remSleep` | 分钟 | 每日汇总 | R04 戒指 |
| 11 | 日常综合 | `Daily` | `dailySteps`, `dailyCalories` | 多种 | 每日汇总 | 所有设备 |

所有数据类型在 `miniprogram/types/healthData.ts` 中以 TypeScript 接口严格定义，继承自 `BaseRecord`（含 ISO 8601 时间戳），确保采集数据的类型安全。

## 五、数据结构化处理流程

```
BLE 原始字节流
    ↓
Veepoo SDK 协议解析（vp_sdk/index.js）
    ↓
TypeScript 类型实例化（healthData.ts 接口校验）
    ↓
JSON Schema 结构化校验
    ├── 字段完整性检查（必填字段不为空）
    ├── 值域校验（心率 30-250 bpm、血氧 70-100%、血压 60-260 mmHg）
    └── 单位标准化（体温 °F→°C、血糖 mg/dL→mmol/L）
    ↓
FHIR Resources 映射
    ├── 心率/血氧/血压/体温/ECG/血糖/睡眠/步数 → FHIR Observation
    └── 结构化报告（日报/周报） → FHIR DiagnosticReport
    ↓
数据去重（同一设备同一时间戳的重复上传过滤）
    ↓
HTTP POST 上传（HTTPS 或 MQTT 通道）
    ↓
MySQL 研究队列数据库（smart_followup_research）
```

### 常用查询命令

**查询 1：研究队列总览**
```sql
SELECT c.cohort_name AS '队列名称', c.principal_investigator AS '负责人',
       c.status AS '状态', c.target_size AS '目标样本量',
       COUNT(DISTINCT p.id) AS '已入组受试者',
       COUNT(DISTINCT d.id) AS '绑定设备数'
FROM research_cohorts c
LEFT JOIN patients p ON p.cohort_id = c.id
LEFT JOIN medical_devices d ON d.patient_id = p.id
GROUP BY c.id;
```

**查询 2：二类医疗器械设备清单**
```sql
SELECT d.device_name AS '设备名称', d.model AS '型号',
       d.device_type AS '类型', d.registration_cert_no AS '医疗器械注册证编号',
       d.mac_address AS 'MAC地址', p.name AS '绑定患者'
FROM medical_devices d
JOIN patients p ON d.patient_id = p.id
ORDER BY d.device_name;
```

**查询 3：受试者心率趋势（14 天）**
```sql
SELECT DATE(recorded_at) AS '日期', COUNT(*) AS '采集次数',
       ROUND(AVG(heart_rate),1) AS '平均心率',
       MIN(heart_rate) AS '最低', MAX(heart_rate) AS '最高'
FROM vital_heart_rate WHERE patient_id = 1
GROUP BY DATE(recorded_at) ORDER BY 日期;
```

**查询 4：数据同步日志（含 FHIR 校验状态）**
```sql
SELECT p.name AS '患者', d.device_name AS '设备',
       sl.sync_channel AS '通道', sl.records_count AS '记录数',
       IF(sl.fhir_validated,'通过','未通过') AS 'FHIR校验',
       sl.status AS '状态',
       DATE_FORMAT(sl.synced_at,'%m-%d %H:%i') AS '时间'
FROM data_sync_logs sl
JOIN patients p ON sl.patient_id = p.id
JOIN medical_devices d ON sl.device_id = d.id
ORDER BY sl.synced_at DESC LIMIT 10;
```

**查询 5：数据库总量统计**
```sql
SELECT
  (SELECT COUNT(*) FROM research_cohorts) AS '研究队列',
  (SELECT COUNT(*) FROM patients) AS '受试者',
  (SELECT COUNT(*) FROM medical_devices) AS '设备',
  (SELECT COUNT(*) FROM vital_heart_rate) +
  (SELECT COUNT(*) FROM vital_blood_oxygen) +
  (SELECT COUNT(*) FROM vital_blood_pressure) +
  (SELECT COUNT(*) FROM vital_signs) AS '体征记录总数',
  (SELECT COUNT(*) FROM data_sync_logs) AS '同步日志',
  (SELECT COUNT(*) FROM structured_reports) AS '分析报告';
```

### BLE 连接与数据采集代码示例

```typescript
import VPManager from '../../libs/vp_sdk/index';

// 1. 扫描 BLE 设备
VPManager.startScanDevice((device) => {
  console.log('发现设备:', device.name, device.mac);
});

// 2. 连接设备（通过 MAC 地址）
VPManager.connectDevice(macAddress, (state) => {
  if (state === 'connected') {
    // 3. 密钥认证（必须在采集数据前完成）
    VPManager.sendSecretKey(key, (result) => {
      console.log('认证成功，可以开始采集数据');
    });
  }
});

// 4. 采集心率数据
VPManager.startDetectingHeartRate((data) => {
  const record: HeartRate = {
    timestamp: new Date().toISOString(),
    heartRate: data.heartRate,
    heartState: data.state  // resting/active/sleeping/exercise
  };
  // 保存到本地并自动同步到服务端
  DataStorageService.getInstance().saveHeartRate(record);
});

// 5. 采集血氧数据
VPManager.startDetectingBloodOxygen((data) => {
  const record: BloodOxygen = {
```

## 九、注意事项

1. **蓝牙测试需真机**：微信开发者工具模拟器不支持 BLE 通信，所有设备功能必须在真机上测试
2. **密钥认证**：连接设备后必须完成密钥认证才能调用数据采集 API，否则会返回空数据
3. **框架限制**：微信小程序 SDK 仅支持原生 + TypeScript 开发，不兼容 mpvue、uni-app、Taro 等框架
4. **数据合规**：采集的体征数据涉及个人健康信息，使用时需遵守《个人信息保护法》和《医疗数据安全管理办法》
5. **设备电量**：长时间连续采集（如 24 小时心率监测）会显著消耗设备电量，建议设置合理的采集间隔
6. **数据库备份**：研究队列数据库应定期备份，`database/init.sql` 可用于完整恢复
